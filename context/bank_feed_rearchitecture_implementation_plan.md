# Bank Feed Rearchitecture – Implementation Plan

## 0. Scope & Goals
- Move bank-feed transaction storage out of `apps.plaid` into a new generic `apps.bank_feed` app.
- Support multiple ingestion sources (Plaid, CSV, manual) behind one model and one categorization workflow.
- Keep the ledger (`JournalEntry`/`JournalLine`) clean and authoritative.
- Preserve the existing Plaid transactions API behavior and bank feed UI while preparing for CSV import.

## 1. Create `apps.bank_feed` app and core model (user steps 1–2)
1. Create new Django app `apps.bank_feed` and add it to `INSTALLED_APPS`.
2. In `apps.bank_feed.models`, define `ImportedTransaction` (extends `BaseTeamModel`) by copying all fields from `apps.plaid.models.ImportedTransaction`, including the FK to `journal.JournalEntry`.
3. Add a new `source` field on `ImportedTransaction`:
   - `CharField(max_length=20, choices=[("plaid", "Plaid"), ("csv", "CSV"), ("manual", "Manual")], default="plaid")`.
   - Document that Plaid-created rows always have `source="plaid"`, CSV uploads use `"csv"`, and directly keyed bank-feed rows use `"manual"`.
4. For now, keep the Plaid-specific fields (`plaid_transaction_id`, `plaid_account`, etc.) on this model and note that they are only populated when `source="plaid"`.
5. Create initial migrations for `bank_feed` (schema only).

## Wipe the database
1. Wipe the database and re-run migrations to ensure a clean starting point.


## 3. Point existing Plaid flows at the new model (maintain current behavior)
1. Update imports so all backend code that currently uses `apps.plaid.models.ImportedTransaction` instead imports from `apps.bank_feed.models`:
   - `apps.plaid.tasks` (`process_added_transaction`, `process_modified_transaction`, `process_removed_transaction`).
   - `apps.plaid.views.ImportedTransactionViewSet`.
   - `apps.plaid.serializers.ImportedTransactionSerializer` and `imported_tx_to_feed_row`.
2. Move `ImportedTransactionSerializer` into `apps.bank_feed.serializers`, then update Plaid viewsets to import it from `apps.bank_feed`.
4. After all references have been updated and tests pass, deprecate the old `apps.plaid.models.ImportedTransaction` class and eventually remove it in a follow-up migration.

## 4. New canonical serializer & API for imported transactions (user steps 3–4)
1. In `apps.bank_feed.serializers`, create a canonical `ImportedTransactionSerializer` for the new model:
   - Base fields: id, team, amount, date, description/name, merchant, pending, payment_channel, etc.
   - Include `source` and any Plaid-specific fields (for `source="plaid"`).
   - Include `journal_entry` id and a derived boolean `is_categorized` (True iff `journal_entry` is not null).
2. Optionally add a `SerializerMethodField` like `journal_entry_summary` (date, description, primary account/category) to make it easy for the UI to show categorized vs uncategorized states without extra API calls.
3. In `apps.bank_feed.views`, create `ImportedTransactionViewSet`:
   - Use `TeamModelAccessPermissions` and `for_team` manager.
   - Support `list`/`retrieve` (read-only for now) with filters for `account`, `source`, `date_from`/`date_to`, and `is_categorized`.
   - This viewset uses the canonical serializer from step 1.
4. In `apps.bank_feed.urls`, register a router with at least `router.register("transactions", ImportedTransactionViewSet, basename="bank-feed-transaction")`.
5. Mount new URLs under `team_urlpatterns` in `koala_budget/urls.py`, e.g. `/a/<team_slug>/bank-feed/api/transactions/`.
6. Add `drf-spectacular` annotations (`extend_schema_view`, `extend_schema`) for the new viewset so the OpenAPI schema documents all query params and response shapes.
7. Keep Plaid’s `/plaid/api/transactions/` endpoint as a thin wrapper over the new viewset/serializer so existing autogenerated clients continue to work, while the new `/bank-feed/api/transactions/` endpoint becomes the generic entry point for future CSV/manual imports.

## 5. JournalLine model change (user step 5)
1. In `apps.journal.models.JournalLine`:
   - Add a new field `reconciled_date = models.DateField(null=True, blank=True, help_text="Date this line was reconciled to a bank statement")`.
   - Temporarily keep `is_cleared` so the app continues to run.
3. Search for all usages of `is_cleared` (serializers, views, filters, admin, tests):
   - In `BankFeedRowSerializer` / `journal_line_to_feed_row`, change the logic to derive `is_cleared` from `reconciled_date` (e.g., `bool(line.reconciled_date)`) or from `JournalEntry.status`, while keeping the API field name the same for now.
   - Update any filters or reports that currently depend on `is_cleared` to use `reconciled_date` instead.
4. Once all references are updated and tests pass, remove the `is_cleared` field in a separate migration and clean up any dead code.

## 6. Bank feed / categorization API adjustments
1. Update `BankFeedViewSet` to use `apps.bank_feed.ImportedTransaction` instead of the old Plaid model, filtering by:
   - Team, ledger account, and `journal_entry__isnull=True` for uncategorized rows.
   - `source` in {"plaid", "csv", "manual"} as needed.
2. Ensure `imported_tx_to_feed_row` continues to produce the same fields consumed by the frontend (`id`, `source`, `date`, `description`, `account`, `category`, `inflow`, `outflow`, `is_pending`, `is_cleared`, `payment_channel`, `confidence`, `journal_line_id`, `imported_transaction_id`, `is_editable`).
   - For non-Plaid sources, derive `account` directly from the ledger `Account` FK rather than PlaidAccount.
   - For categorized imported transactions (those with `journal_entry` set), either:
     - Exclude them from the "uncategorized" feed and show only via ledger rows, or
     - Represent them as enriched rows by looking up the linked `JournalEntry`.
3. Confirm the `/bank-feed/categorize/` endpoint still works end-to-end:
   - It must handle rows from any `source` value (Plaid/CSV/manual), but the underlying journal-entry creation logic remains the same.
4. in the teams app there is a signal called `bootstrap_team_on_create` that creates a default account and category template.  Update this to use the new `apps.bank_feed` models.

## 7. Frontend updates (user step 6)
1. Review `assets/javascript/journal/react/LineApp.jsx`:
   - Confirm the bank feed is still loaded from `/a/${teamSlug}/plaid/api/bank-feed/?account=...`.
   - Decide whether to keep this URL or, in a later iteration, switch to a more neutral `/bank-feed/` path once backend shims are stable.
2. Review `assets/javascript/journal/react/BankFeedTable.jsx`:
   - Ensure it continues to treat `line.source === 'plaid'` as an imported row; if backend ever changes `source` semantics, introduce a separate field (e.g. `import_source`) in the serializer to distinguish ledger vs bank-feed source vs ingestion type.
   - Update the categorization controls to work generically with bank-feed rows (not Plaid-specific), relying on `is_categorized` / `journal_entry` rather than Plaid-only hints.
3. Regenerate the OpenAPI-based client (if needed) so any new fields (`source`, `is_categorized`, `reconciled_date`, etc.) are available in TypeScript types, then update the React code to use the generated types instead of ad-hoc field accesses.
4. Manually verify in the UI that:
   - Plaid-imported transactions still appear and can be categorized.
   - Categorized transactions show as such in the feed.
   - Nothing regresses in the existing bank feed / journal lines views.

## 8. Testing & rollout
1. Add/extend backend tests for:
   - `bank_feed.ImportedTransaction` model and serializer (including both categorized and uncategorized cases).
   - `/plaid/api/transactions/` and new `/bank-feed/api/transactions/` endpoints.
   - `BankFeedViewSet` combined feed behavior across different `source` values.
   - `JournalLine` reconciliation logic using `reconciled_date`.
2. Add frontend tests (where feasible) around the bank feed table and categorization actions.
3. Run `make migrate`, `make test`, and `make npm-type-check` before deploying.
4. Plan a short migration window to run the data migration from Plaid → bank_feed in production, and keep the old Plaid model around (read-only) until the new paths are fully validated.
